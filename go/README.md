# AntiSpamBotTG (Go)

Telegram-бот для автоматической модерации чатов. Работает как антифлуд: если пользователь присылает больше `RATE_LIMIT_COUNT` сообщений за `RATE_LIMIT_WINDOW_SECONDS`, бот мгновенно удаляет очередное сообщение и банит пользователя с отзывом (удалением) всех его недавних сообщений в чате — одним запросом к Telegram API (`RevokeMessages: true`).

Обработка асинхронная: входящие обновления от Telegram сразу кладутся в очередь и разбираются пулом воркеров, поэтому один медленный вызов Telegram API (удаление/бан) не блокирует приём остальных сообщений. При аномальном всплеске нагрузки, когда очередь переполняется, новые обновления отбрасываются с предупреждением в лог — вместо падения бота или неограниченного роста памяти.

Переписан с нуля на Go (изначальная версия была на Python/python-telegram-bot и использовала чёрный список слов) с прицелом на минимальное потребление ресурсов и работу 24/7 в Docker.

## Пример работы

```
Пользователь: привет
Пользователь: привет
Пользователь: привет
Пользователь: привет
Пользователь: привет
Пользователь: привет   <- 6-е сообщение за 3 секунды

Бот: [6-е сообщение удалено]
Бот: [пользователь забанен, его предыдущие 5 сообщений тоже удалены]
Бот: 🚫 Пользователь @spam_user123 забанен за флуд.
```

Лог бота:
```
2026-07-03 21:00:01 WARN обнаружен флуд, бан пользователя user_id=123 username=spam_user123 chat_id=-100123456
```

## Как получить токен бота и настроить права администратора

1. В Telegram откройте чат с [@BotFather](https://t.me/BotFather) и отправьте `/newbot`.
2. Следуйте инструкциям (имя, username бота) — BotFather пришлёт токен вида `789123:AAH...`. Это значение для `BOT_TOKEN`.
3. Добавьте бота в нужный чат/группу.
4. Откройте настройки группы → «Администраторы» → добавьте бота в администраторы.
5. Обязательно включите боту права **«Удаление сообщений»** (Delete messages) и **«Блокировка пользователей»** (Ban users) — без них бот не сможет удалять сообщения и банить флудеров. Если планируете использовать капчу для новичков (включена по умолчанию) — также включите **«Ограничение участников»** (Restrict members), иначе мьют новых участников не сработает.

## Деплой

1. Склонируйте репозиторий и перейдите в директорию с Go-версией:
   ```
   git clone https://github.com/Hristofor1234/AntiSpamBotTG.git
   cd AntiSpamBotTG/go
   ```
2. Скопируйте файл окружения и заполните своими значениями:
   ```
   cp .env.example .env
   nano .env
   ```
   Обязательно: `BOT_TOKEN` (из BotFather). Остальные переменные (`RATE_LIMIT_COUNT`, `RATE_LIMIT_WINDOW_SECONDS`, `WORKER_COUNT`, `QUEUE_SIZE`, `DATABASE_URL`, `WEBHOOK_*`) необязательны — при отсутствии используются значения по умолчанию (5 сообщений / 3 секунды, 10 воркеров, очередь на 5000, Long Polling, без глобального обучения).
3. Запустите проект одной командой:
   ```
   docker compose up -d --build
   ```
4. Проверьте, что бот поднялся:
   ```
   docker compose logs -f bot
   ```
   В логе должна появиться строка о запуске бота с параметрами антифлуда.
5. Напишите боту `/start` в Telegram — должен прийти ответ с текущими настройками лимита.

## Команды бота в Telegram

`/start` или `/help` — полный список функций бота (автозащита, команды админов и всех пользователей) и текущие настройки (режим приёма обновлений, лимиты, статус PostgreSQL, капчи, тихого режима). Оба показывают один и тот же текст — удобно посмотреть с телефона, что умеет бот, не заглядывая в README.

`/report` — отправляется **в ответ (reply)** на сообщение, которое вы считаете спамом. Подробности — в разделе «Обучение через жалобы» ниже.

`/addspam <фраза>`, `/removespam <фраза>`, `/triggers` — управление фильтром по словам для текущего чата, доступно только администраторам чата. Подробности — в разделе «Фильтр по словам» ниже.

`/blockdomain <домен>`, `/unblockdomain <домен>`, `/domains` — управление глобальным списком опасных доменов, доступно только администраторам чата. Подробности — в разделе «Проверка ссылок по домену» ниже.

`/ban` (**в ответ (reply) на сообщение спамера**) — ручной бан администратором с автообучением. Подробности — в разделе «Ручной бан администратором (/ban)» ниже.

## Приём обновлений: Long Polling или Webhook

По умолчанию бот работает через **Long Polling** (`b.Start`) — ничего дополнительно настраивать не нужно, этого достаточно для большинства чатов.

Webhook имеет смысл включать только при заметном трафике (много чатов/сообщений в секунду) — он снижает задержку и нагрузку на сеть, но требует своей инфраструктуры:

1. Публичный домен с HTTPS-сертификатом (Telegram принимает только `https://` в `setWebhook`).
2. Reverse-proxy (nginx, Caddy и т.п.), который терминирует TLS и проксирует на порт бота (`WEBHOOK_LISTEN_ADDR`, по умолчанию `:8080`) — сам бот отдаёт обычный HTTP, сертификатами не занимается.

Чтобы включить: задайте в `.env` `WEBHOOK_URL` (например, `https://example.com/webhook`) и `WEBHOOK_SECRET_TOKEN` (случайная строка — Telegram присылает её в заголовке `X-Telegram-Bot-Api-Secret-Token`, бот сверяет и отбрасывает запросы без неё или с неверной). При старте бот сам вызывает `setWebhook`; при остановке — снимает его (`deleteWebhook`) и корректно завершает HTTP-сервер (graceful shutdown).

Если `WEBHOOK_URL` не задан — бот работает через Long Polling, как и раньше.

## Устойчивость к 429 Too Many Requests

Вызовы `deleteMessage`, `banChatMember` и `sendMessage` обёрнуты в ретрай (`internal/dispatcher/retry.go`): если Telegram отвечает 429 (например, бот банит много флудеров подряд), бот ждёт ровно `retry_after` секунд, которые вернул Telegram, и повторяет попытку — до 3 раз, — вместо того чтобы просто залогировать ошибку и потерять бан.

## Глобальное обучение (PostgreSQL)

Помимо rate-limit, у бота есть второй, более быстрый уровень обороны — глобальный чёрный список спама в PostgreSQL (`internal/storage`). Работает так:

1. Когда бот банит пользователя за флуд, текст его сообщения нормализуется (нижний регистр, схлопывание пробелов, удаление невидимых/управляющих юникод-символов — частый приём спамеров, чтобы обойти точное сравнение текста) и его хэш (SHA-256) сохраняется в таблице `global_blacklist`.
2. При получении любого следующего сообщения — в этом же чате или в любом другом, где стоит бот, — текст точно так же нормализуется и проверяется по этой таблице. При совпадении хэша пользователь банится мгновенно, ещё до срабатывания rate-limit — не нужно ждать, пока он повторит спам несколько раз в конкретном чате.

Фича полностью необязательная и безопасно деградирует:

- Если `DATABASE_URL` не задан — бот работает как раньше, только на in-memory rate-limit.
- Если `DATABASE_URL` задан, но PostgreSQL недоступен при старте (после нескольких попыток подключения) — бот всё равно запускается, просто без глобального обучения, и пишет об этом в лог.
- Если PostgreSQL "отваливается" уже во время работы — ошибка проверки логируется, а сообщение всё равно проходит через rate-limit как обычно.

Чтобы включить: поднимите PostgreSQL (в `docker-compose.yml` уже есть готовый сервис `db` — если он не нужен, секцию можно удалить) и задайте в `.env`:
```
DATABASE_URL=postgres://antispam:antispam@db:5432/antispam?sslmode=disable
```
Таблицы создаются автоматически при старте (миграция идемпотентна, ничего вручную выполнять не нужно).

## Проверка ссылок по домену (Link Checker)

Ещё один уровень глобального обучения (`internal/linkcheck` + таблица `bad_domains` в PostgreSQL) — но не по тексту целиком, а по домену ссылки. Сознательно сделан **без внешних сервисов** (VirusTotal, Google Safe Browsing): у обоих либо жёсткие бесплатные лимиты (VirusTotal — 4 запроса/мин), либо требуется отдельный API-ключ и сетевой вызов на каждое сообщение со ссылкой. Вместо этого используется тот же принцип, что и у глобального текстового обучения:

1. Когда бот банит пользователя (за флуд, за жалобы или по триггер-словам) и в его сообщении была ссылка — из неё извлекается домен (без `www.`, без протокола, без пути) и сохраняется в `bad_domains`.
2. Любое следующее сообщение с ссылкой на уже известный опасный домен — в этом же чате или в любом другом, где стоит бот — банится мгновенно, даже если сам текст сообщения новый и раньше не встречался.

Список доменов глобальный (как и `global_blacklist`), поэтому админы могут управлять им из любого чата, где есть бот:

- `/blockdomain <домен>` — добавить домен в глобальный список вручную (например, если спам заметили раньше, чем бот успел кого-то забанить).
- `/unblockdomain <домен>` — убрать домен (например, ложное срабатывание).
- `/domains` — последние 30 доменов в списке.

Требует подключённой PostgreSQL (`DATABASE_URL`) — без неё уровень проверки доменов и команды выше просто отключены, как и остальные фичи на БД.

Ограничения такого подхода стоит понимать: домен должен сначала где-то "засветиться" в забаненном сообщении (или быть добавлен вручную), прежде чем бот начнёт его блокировать — это не проверка репутации домена по внешней базе, а обучение на собственном опыте бота и админов. Проверка репутации через Google Safe Browsing (у него более щедрый бесплатный лимит, ~10 тыс. запросов/день) может быть добавлена поверх этого уровня отдельно.

## Ручной бан администратором (/ban)

Все предыдущие уровни обороны реагируют либо на повтор (rate-limit, глобальный ЧС), либо на голосование участников (`/report`), либо на заранее известные фразы/домены. `/ban` закрывает случай, когда админ **сразу видит** спам, который бот сам ещё не отловил — например, ссылку на только что зарегистрированный домен, разовое сообщение без флуда, или спам, до которого `/report` ещё не набрал кворум.

Как использовать: ответить (reply) на сообщение спамера командой `/ban`. Бот:

1. Проверяет, что отправитель команды — админ или создатель чата.
2. Банит автора сообщения (с `RevokeMessages: true` — удаляются все его недавние сообщения в чате), тем же путём, что и автоматический бан за флуд (`internal/dispatcher.BanSpammer`).
3. Если подключена PostgreSQL — добавляет текст сообщения в `global_blacklist` и домены из ссылок в нём (если есть) в `bad_domains`. Дальше такой же спам (или ссылка на тот же домен) будет отсекаться мгновенно в любом чате, где стоит бот — без ожидания повторного флуда или новых жалоб.

Ограничения: нельзя ответить `/ban` на сообщение бота или на собственное сообщение. Уведомление о бане в чат отправляется (или не отправляется, если `SILENT_BAN=true`) тем же механизмом, что и у автоматических банов — отдельного сообщения от `/ban` не будет, чтобы не дублировать.

## Обучение через жалобы (/report)

Третий источник спам-сигнала, дополняющий rate-limit и глобальный чёрный список — коллективное мнение участников чата. Работает так:

1. Кто-то отвечает (reply) на подозрительное сообщение командой `/report`.
2. Бот засчитывает жалобу от этого пользователя (`internal/reports`). Повторная жалоба от одного и того же человека не учитывается дважды.
3. Как только на сообщение пожалуются `REPORT_THRESHOLD` разных пользователей (по умолчанию 3) — бот удаляет сообщение, банит автора с отзывом недавних сообщений (то же самое, что при бане за флуд) и, если подключена PostgreSQL, добавляет текст в глобальное обучение — дальше такой же спам будет отсекаться мгновенно в любом чате.

Ограничения, о которых стоит знать:
- Пожаловаться на собственное сообщение нельзя.
- Жалобы на сообщения от других ботов игнорируются.
- Незакрытые жалобы (порог так и не набрался) забываются через 24 часа — чтобы не копить память бесконечно.
- Это текстовая команда, а не inline-кнопка «Это спам» под сообщением: Telegram не позволяет боту добавлять кнопки к чужим сообщениям, а плодить отдельное сообщение-кнопку под каждым сообщением в чате было бы избыточным спамом само по себе. `/report`-реплай — обычная практика модераторских ботов и не требует прав, кроме отправки сообщений.

Порог настраивается через `REPORT_THRESHOLD` в `.env` (минимум 2, по умолчанию 3).

## Фильтр по словам (/addspam, /removespam, /triggers)

Свой список запрещённых фраз для каждого чата — в отличие от глобального обучения (общее для всех чатов, наполняется автоматически), этот список локальный и наполняется вручную администраторами конкретного чата. Требует подключённой PostgreSQL (`DATABASE_URL`) — без неё команды отвечают, что фильтр недоступен, вместо того чтобы молча ничего не делать.

Как работает:

1. Администратор чата добавляет фразу: `/addspam казино`. Сравнение идёт по нормализованному тексту (регистр, пробелы, невидимые символы игнорируются — как и в глобальном обучении), так что `/addspam Казино` и `/addspam  казино ` считаются одной и той же фразой.
2. Когда кто-то в чате присылает сообщение, содержащее эту фразу (как подстроку, после нормализации), бот удаляет сообщение и выдаёт предупреждение: `⚠️ ...: предупреждение 1/3 за запрещённое слово`.
3. После `WARN_THRESHOLD` предупреждений подряд (по умолчанию 3) — бан вместо предупреждения, с тем же обучением глобального чёрного списка, что и при бане за флуд.

`/removespam <фраза>` убирает фразу из фильтра, `/triggers` показывает текущий список фраз чата. Все три команды проверяют, что отправитель — админ или создатель чата (`getChatMember`), иначе отвечают отказом.

Проверка триггер-слов идёт **после** rate-limit и глобального чёрного списка (см. `internal/dispatcher`) — это самый "дорогой" по сути уровень (нужно ручное наполнение), поэтому он последний.

## Тихий режим (SILENT_BAN)

По умолчанию после бана (за флуд, за жалобы или за триггер-слова) бот пишет в чат `🚫 Пользователь X забанен за спам.`, а при накоплении предупреждений — `⚠️ ...: предупреждение N/M`. Если это не нужно (например, чат не хочет видеть "публичные казни"), задайте `SILENT_BAN=true` — сообщения-уведомления перестанут отправляться. Само удаление сообщений, бан и обучение при этом продолжают работать как обычно — меняется только то, что видно в чате.

## Капча для новых участников чата

Третий, "нулевой" уровень обороны — не от спама в сообщениях, а от ботов и накрученных аккаунтов, вступающих в чат. Работает по событию вступления (`new_chat_members` от Telegram), а не по эвристике вроде "первое сообщение" или "аккаунт младше N дней" — Telegram Bot API вообще не отдаёт дату регистрации аккаунта, а привязка к "первому сообщению от пользователя" сломалась бы после каждого перезапуска бота, заново считая всех давних участников чата новичками.

Как работает:

1. Пользователь вступает в чат → бот сразу ограничивает ему права (`restrictChatMember`, полный мьют) и присылает сообщение с кнопкой «✅ Я не бот».
2. Нажал кнопку — бот возвращает ему права по умолчанию для этого чата (те, что реально настроены в чате, не жёстко зашитый набор) и удаляет сообщение с капчей.
3. Не нажал за `CAPTCHA_TIMEOUT_SECONDS` (по умолчанию 120) — бот удаляет его из чата (кик с немедленным снятием бана, то есть человек сможет зайти и попробовать снова, а не заблокирован навсегда).

Кнопку может нажать только тот, кого капчуют (проверяется `user_id` нажавшего) — иначе кто угодно в чате мог бы подтвердить за другого. Отключается через `CAPTCHA_ENABLED=false`. Требует у бота права **Restrict members** — без него `restrictChatMember` будет возвращать ошибку (см. лог).

## Подсказки команд в Telegram (setMyCommands)

При старте бот регистрирует список команд для автодополнения — при вводе `/` в чате Telegram показывает подсказки с описанием. Список разный для групп и личных чатов:

- В группах — полный список (все команды, включая административные). Видимость подсказки не ограничивает права: `/ban`, `/addspam` и остальные админские команды всё равно проверяют, что отправитель — админ чата (см. выше), не подтвердившим права просто откажут при попытке выполнить, а не скроют команду из подсказки для всех.
- В личных чатах — только `/start` и `/help`: остальные команды либо требуют reply на сообщение в группе (`/ban`, `/report`), либо управляют настройками конкретного чата (`/addspam` и т.д.), поэтому в личке бесполезны.

## Автоудаление команд и ответов в группах

Чтобы админские команды не оставались навсегда в истории чата, оба сообщения — сама команда (например, `/addspam казино`) и ответ бота — удаляются через `AUTO_DELETE_DELAY_SECONDS` (по умолчанию 30 секунд). Работает для `/start`, `/help`, `/report`, `/addspam`, `/removespam`, `/triggers`, `/blockdomain`, `/unblockdomain`, `/domains` — реакции на реальные команды администраторов и пользователей.

Важные нюансы:

- В личных чатах ничего не удаляется — там это обычная переписка, а не разовая команда в общем чате.
- Автоматические уведомления о банах за флуд/жалобы/триггер-слова (см. разделы выше) автоудалением пока не затронуты — это отдельный механизм (см. `SILENT_BAN`, если не нужны вообще).
- `AUTO_DELETE_DELAY_SECONDS=0` полностью отключает автоудаление.
- Требует у бота права **«Удаление сообщений»** (Delete messages) — оно уже нужно для основной антифлуд-функции бота, отдельно включать ничего не надо.

## Архитектура проекта

```
AntiSpamBotTG/go/
├── cmd/bot/main.go            # точка входа, связывает конфиг, лимитер, БД, диспетчер и Telegram-бота
├── cmd/bot/webhook.go         # HTTP-сервер webhook: setWebhook/deleteWebhook, graceful shutdown
├── cmd/bot/admin.go           # команды /addspam, /removespam, /triggers + проверка прав администратора
├── cmd/bot/captcha.go         # капча для новых участников: мьют, кнопка, снятие ограничений/кик
├── internal/config/           # загрузка и валидация конфигурации из окружения
├── internal/ratelimit/        # потокобезопасный счётчик сообщений на пользователя + периодическая очистка
├── internal/storage/          # PostgreSQL: глобальный ЧС, опасные домены, триггер-слова чатов, счётчики предупреждений
├── internal/linkcheck/        # извлечение доменов из ссылок в тексте (без внешних API)
├── internal/reports/          # обучение через жалобы: уникальные жалобщики на сообщение + TTL-очистка
├── internal/captcha/          # ожидание прохождения капчи новыми участниками (таймауты)
├── internal/dispatcher/       # пул воркеров: уровни обороны (ЧС → опасные домены → rate-limit → триггер-слова → обучение)
├── internal/dispatcher/retry.go  # ретраи вызовов Telegram API при 429 Too Many Requests
├── Dockerfile                  # multi-stage build, итоговый образ на alpine
└── docker-compose.yml          # сервис bot + опциональный сервис db (PostgreSQL)
```

Счётчики rate-limit по-прежнему живут только в памяти процесса и сбрасываются при перезапуске — это осознанное упрощение, не требующее БД для базовой защиты от флуда. Персистентен только глобальный чёрный список (при подключённой PostgreSQL).
