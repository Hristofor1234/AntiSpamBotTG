# Отчёт: пересборка AntiSpamBotTG на Go и деплой на сервер

Дата: 3 июля 2026 г.

## Исходная задача

Проект AntiSpamBotTG изначально представлял собой Telegram-бота на Python (python-telegram-bot 20.7, psycopg2, PostgreSQL), развёрнутого через вебхук на Railway/Heroku. Бот удаляет сообщения, содержащие слова из чёрного списка, а администратор управляет списком через команды `/add`, `/remove`, `/list`, `/commands`. Доступ к командам был ограничен одним хардкоженным username.

По техническому заданию требовалось переписать бота на Go с сохранением функциональности 1:1, добавить кэширование чёрного списка в памяти для производительности, контейнеризировать проект через Docker и подготовить его к развёртыванию 24/7 с минимальным потреблением ресурсов.

## Реализация на Go

Проект собран по слоистой структуре: точка входа `cmd/bot/main.go`, а бизнес-логика разнесена по пакетам `internal/config`, `internal/database`, `internal/handler` и `internal/middleware`.

Пакет `config` отвечает за чтение и валидацию переменных окружения (`BOT_TOKEN`, `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`, `ALLOWED_USERS`); при отсутствии обязательного значения бот завершается с понятной ошибкой в логе, как и требовало ТЗ. Список администраторов парсится в множество (`map[string]struct{}`) и проверяется без учёта регистра.

Пакет `database` инкапсулирует пул соединений `pgxpool` (5–10 соединений, таймауты подключения) и потокобезопасный in-memory кэш чёрного списка на `sync.RWMutex`. Проверка сообщения на спам (`ContainsSpam`) идёт исключительно по кэшу, без обращения к БД на каждое сообщение — это было явным требованием ТЗ по производительности. Кэш перечитывается из БД при старте и обновляется точечно при выполнении команд `/add` и `/remove`, оборачиваемых в транзакции.

Пакет `middleware` реализует проверку доступа (ACL): обработчик оборачивается функцией `RequireAccess`, которая сверяет username отправителя со списком `ALLOWED_USERS` и в случае отказа отвечает `⛔ У вас нет доступа.`, не выполняя команду. Команда `/start` сознательно оставлена публичной (доступна всем), остальные четыре команды защищены middleware — это соответствует поведению исходного Python-бота, где ACL стоял только на управляющих командах.

Пакет `handler` содержит обработчики команд (`Start`, `Help`, `Add`, `Remove`, `List`) и обработчик фильтра спама по умолчанию (`Spam.Filter`), который проверяет текст входящего сообщения по кэшу и мгновенно удаляет его при совпадении, игнорируя сообщения от самого бота и от других ботов. Для маршрутизации команд использована библиотека `github.com/go-telegram/bot`, точная сигнатура API которой (типы `HandlerFunc`, `Middleware`, `RegisterHandler`, `DeleteMessageParams` и структура `models.Message`) была сверена по официальному README и исходному коду библиотеки на GitHub, а не восстановлена по памяти. Бот работает через long polling (`b.Start`), а не через вебхук, как в Python-версии, — это упростило деплой: не требуется публичный домен, TLS-сертификат и открытый порт для входящих запросов.

Для контейнеризации подготовлен `Dockerfile` с двухэтапной сборкой (`golang:1.22-alpine` → `alpine:3.19`), запуском от непривилегированного пользователя, и `docker-compose.yml` с двумя сервисами — `db` (postgres:15-alpine, healthcheck через `pg_isready`) и `bot`. Дополнительно подготовлены `.env.example`, `.gitignore`, `.dockerignore` и подробный `README.md` с инструкцией по деплою, получению токена бота и настройке прав администратора в чате.

## Ограничения проверки кода

Рабочая песочница, в которой велась разработка, не имеет установленного компилятора Go и не имеет сетевого доступа к proxy.golang.org, поэтому автоматический прогон `go build` / `go vet` оказался невозможен. Вместо этого код был проверен вручную: сверена балансировка скобок во всех файлах, а сигнатуры используемых функций и структур библиотек `go-telegram/bot` и `jackc/pgx/v5` сверены построчно с их актуальными исходниками на GitHub. Фактическая компиляция впервые происходит на стороне Docker при сборке образа (`go mod tidy` в Dockerfile сам скачивает зависимости и достраивает `go.sum`, так как закоммитить его заранее без доступа к модульному прокси было невозможно).

## Публикация в GitHub

Готовый код был закоммичен в локальный репозиторий и должен был быть отправлен на `github.com/Hristofor1234/AntiSpamBotTG`. Прямой `git push` из рабочей среды оказался невозможен по той же причине отсутствия сетевого доступа к GitHub из песочницы, поэтому пуш выполнялся пользователем со своего компьютера.

При первой попытке пуша GitHub вернул `403: Permission denied`, хотя аккаунт был правильным. Причина оказалась в устаревших/некорректных учётных данных, закэшированных в macOS Keychain: попытка `git credential-osxkeychain erase` завершилась ошибкой `failed to erase: -1`, то есть штатная очистка не сработала. Проблема была диагностирована через прямой пуш с токеном, встроенным в URL (обошли Keychain), что подтвердило: сам токен рабочий, а мешает именно закэшированная запись. Запись была принудительно удалена через системную утилиту `security delete-internet-password -s github.com`, после чего обычный `git push origin master` отработал штатно с запросом логина и токена в терминале.

В процессе пользователь по неосторожности вставил реальный Personal Access Token в открытый чат — было рекомендовано немедленно отозвать этот токен на странице `github.com/settings/tokens` и в дальнейшем не передавать токены в открытом виде. Также был согласован минимально достаточный набор прав для нового токена — только чекбокс `repo` (Full control of private repositories), без лишних прав вроде `admin:org` или `delete_repo`.

## Деплой на сервер

Для развёртывания использовался уже существующий VPS пользователя: Ubuntu 24.04.4 LTS, Docker 29.6.1, Docker Compose v5.2.0 — оба уже были установлены, что избавило от необходимости использовать `sudo` при развёртывании. Сервер доступен только по частному IP-адресу локальной сети (`192.168.57.26`), поэтому прямое SSH-подключение из рабочей среды технически невозможно (не имеет маршрута из облачной песочницы); все команды на сервере пользователь выполнял самостоятельно по присланным инструкциям, результат вывода которых анализировался для планирования следующего шага.

Перед развёртыванием была изучена структура сервера во избежание конфликтов с уже работающим там проектом `finance-tracker` (сервисы `finance-tracker-bot`, `finance-tracker-api`, `finance_db` на postgres:16-alpine, туннель через `cloudflared`, сеть `finance-tracker-net` и volume `finance-tracker_postgres_data`, порты 8000 и 5432 которого опубликованы только на `127.0.0.1`). Поскольку антиспам-бот работает по long polling и не публикует ни один порт на хост (ни для самого бота, ни для его БД), пересечения портов исключены в принципе. Для дополнительной гарантии изоляции сети и volume в `docker-compose.yml` было явно прописано имя проекта `name: antispam-bot`, чтобы Docker Compose не зависел от имени директории при генерации имён сети и тома.

Проект был склонирован пользователем в `/opt/apps/AntiSpamBotTG` (директория `/opt/apps` уже использовалась под `finance-tracker` и оказалась доступна для записи без sudo). Пользователь настроил `.env`, приняв решение использовать для тестового развёртывания отдельного, специально созданного через @BotFather Telegram-бота — это осознанно исключает конфликт `409 Conflict`, который возник бы при одновременном опросе (`getUpdates`) одного и того же токена старым Python-ботом и новым Go-ботом.

## Текущий статус

На момент составления отчёта пользователь находится на шаге запуска `docker compose up -d --build` в директории `/opt/apps/AntiSpamBotTG/go`. Подтверждение успешной сборки образа, поднятия контейнеров `db` и `bot` и получения ответа от бота на команду `/start` в Telegram — ожидается следующим шагом.
